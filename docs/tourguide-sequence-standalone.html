Chapter 01

Merchant onboarding và session lifecycle

Session không chỉ là JWT phát ra rồi thôi; backend còn kiểm tra sessionId còn active ở mỗi request xác thực.

sequenceDiagram
    actor Merchant
    participant Portal as Merchant Portal
    participant API as AuthController/AuthService
    participant Mongo as MongoDB
    participant Audit as AuditService

    Merchant->>Portal: Nhập họ tên, phone, email, password
    Portal->>API: POST /api/auth/merchant/register
    API->>Mongo: Check phone exists?
    alt Phone đã tồn tại
        API-->>Portal: 400 duplicate merchant
        Portal-->>Merchant: Báo lỗi tạo tài khoản
    else Phone hợp lệ
        API->>Mongo: Insert User(role=Merchant)
        API->>Mongo: Insert UserSession(sessionId)
        API->>Audit: Write MERCHANT_REGISTERED
        API-->>Portal: token + user + sessionId
        Portal->>Portal: Lưu authToken vào local storage
        Portal-->>Merchant: Redirect sang màn tạo POI
    end

    Merchant->>Portal: Mở lại portal sau đó
    Portal->>API: GET /api/auth/me với Bearer token
    API->>Mongo: Read UserSession by sessionId
    alt Session revoked/expired
        API-->>Portal: Unauthorized
        Portal->>Portal: Xóa token local
    else Session active
        API->>Mongo: Update LastSeenAt cho session
        API-->>Portal: SessionUserResponse
        Portal-->>Merchant: Vào dashboard
    end
Điểm hay của flow này là logout không chỉ xóa token client-side. Nó revoke luôn `UserSession`, nên JWT cũ cũng mất hiệu lực ở tầng server.
Chapter 02

Tạo draft POI, translation preview và moderation

Đây là luồng tạo nội dung thực sự của merchant. Nó gộp UI biên tập, translation preview, moderation quyết định và lưu xuống Mongo.

Điểm nhấn nghiệp vụ

  • Merchant luôn tạo POI mới ở package Basic.
  • Translation preview chạy sớm ở UI để merchant chỉnh tay nếu cần.
  • Backend tiếp tục refresh translation cache một lần nữa khi create/update thật.
  • Moderation có thể trả `Rejected`, `Approved` hoặc `PendingManual`.

Ràng buộc dữ liệu

  • Radius effective lấy từ package: Basic 50m, Premium 120m, Boost 200m.
  • Tags nếu rỗng sẽ được suy luận từ tên quán + source description.
  • Merchant update một POI đã review sẽ làm POI quay lại trạng thái draft.
sequenceDiagram
    actor Merchant
    participant Portal as Merchant Register/Manage UI
    participant Translation as TranslationController/Service
    participant API as POIController/PoiService
    participant Moderation as ModerationService
    participant Mongo as MongoDB
    participant Cache as TranslationCache

    Merchant->>Portal: Nhập source description
    Portal->>Translation: POST /api/translation/preview
    alt Translation provider configured
        Translation-->>Portal: Ready translations EN/KO/JA/ZH
    else Provider missing/error
        Translation-->>Portal: PendingManual + error message
    end

    Merchant->>Portal: Chỉnh tay translations và vị trí
    Merchant->>Portal: Bấm Tạo draft
    Portal->>API: POST /api/poi
    API->>Moderation: Evaluate(name + source + merchant note)
    Moderation-->>API: Approved / Rejected / PendingManual
    API->>Mongo: Insert POI(Status=Draft, Package=Basic)
    API->>API: Refresh translations for persisted POI
    API->>Cache: Upsert translation cache by language + contentVersion
    API->>Mongo: Update POI TranslationStatus
    API-->>Portal: PoiListItemResponse
    Portal-->>Merchant: Báo tạo draft thành công
Chapter 03

Submit và review bằng snapshot diff

Luồng review của admin khá xịn: hệ thống không chỉ hiển thị POI mới nhất, mà còn dựng bảng thay đổi so với version approved gần nhất.

sequenceDiagram
    actor Merchant
    participant MerchantUI as Merchant Portal
    participant API as POIController/PoiService
    participant Mongo as MongoDB
    actor Admin
    participant AdminUI as Admin Approval UI
    participant Rules as BusinessRules

    Merchant->>MerchantUI: Bấm Gửi duyệt
    MerchantUI->>API: POST /api/poi/{id}/submit
    API->>Mongo: Update POI Status=Submitted
    API-->>MerchantUI: Submitted OK

    Admin->>AdminUI: Mở màn POI chờ duyệt
    AdminUI->>API: GET /api/poi/submitted-with-changes
    API->>Mongo: Load Submitted POIs
    API->>Mongo: Load latest PoiReviewSnapshot theo poiId
    API->>Rules: BuildPoiApprovalChanges(snapshot, poi)
    API-->>AdminUI: Item + change list + IsNewPoi

    Admin->>AdminUI: Approve hoặc Reject
    AdminUI->>API: POST /api/poi/{id}/review
    API->>Rules: ApplyReview(approve/reject)
    alt Approve
        API->>Mongo: Replace POI(Status=Approved)
        API->>Rules: CreateApprovedSnapshot(poi)
        API->>Mongo: Insert snapshot mới
    else Reject
        API->>Mongo: Replace POI(Status=Rejected, reason)
    end
    API-->>AdminUI: POI đã cập nhật
Snapshot review là mấu chốt giúp admin thấy “merchant vừa sửa gì”, thay vì phải đọc lại toàn bộ hồ sơ bằng mắt.
Chapter 04

Checkout package và kích hoạt sau webhook

Package không đổi ngay lúc bấm mua. Trạng thái POI chỉ được nâng sau khi webhook PayOS xác nhận thanh toán hợp lệ.

sequenceDiagram
    actor Merchant
    participant Portal as Merchant Portal
    participant Billing as BillingController/BillingService
    participant Mongo as MongoDB
    participant PayOS as PayOS Gateway

    Merchant->>Portal: Chọn Premium hoặc Boost
    Portal->>Billing: POST /api/billing/package/checkout
    Billing->>Mongo: Load POI
    alt Package cùng loại còn hiệu lực
        Billing-->>Portal: 400 package already active
    else Hợp lệ
        Billing->>Mongo: Insert BillingRecord(PendingPayment)
        Billing->>PayOS: Create payment link
        PayOS-->>Billing: checkoutUrl + paymentLinkId + orderCode
        Billing->>Mongo: Update BillingRecord checkout fields
        Billing-->>Portal: CheckoutUrl
        Portal-->>Merchant: Mở tab thanh toán
    end

    PayOS->>Billing: POST /api/billing/webhook/payos
    Billing->>PayOS: Verify webhook signature
    alt Signature invalid
        Billing-->>PayOS: ignore / reject
    else Success
        Billing->>Mongo: Mark BillingRecord Active + PaidAt
        alt PackageName = Boost
            Billing->>Mongo: Set POI BoostPriority/BoostExpiresAt/Radius=200
        else PackageName = Premium
            Billing->>Mongo: Set POI SubscriptionPackage=Premium, SubscriptionExpiry, Radius=120
        end
        Billing-->>PayOS: received=true
    end
Premium và Boost có hai rule chống mua trùng riêng: Premium đang active thì không mua lại Premium, Boost còn hạn thì không mua thêm Boost.
Chapter 05

Visitor WebQR: scan counted, narration replay-safe

WebQR tự động tạo visitor/session identity trong browser và chủ động ghi nhận scan ngay khi trang detail được mở.

sequenceDiagram
    actor Visitor
    participant Browser as WebQR page
    participant API as API (POI + Analytics)
    participant Mongo as MongoDB

    Visitor->>Browser: Mở QR URL hoặc /?id=poiId
    Browser->>Browser: Redirect sang /poi/{id}
    Browser->>API: GET /api/poi/details/{id}
    API->>Mongo: Load approved POI + public localized content
    API-->>Browser: Poi detail payload

    Browser->>Browser: ensure visitorId cookie + sessionId storage
    Browser->>API: POST /api/qr/scan(trigger=WebQR)
    API->>Mongo: Find last counted QR by poi + visitor
    alt Ngoài cooldown 6 giờ
        API->>Mongo: Insert counted QrScanLog
        API->>Mongo: Increment POI.CountedQrScanCount
        API-->>Browser: Counted=true
    else Trong cooldown
        API->>Mongo: Insert replay QrScanLog
        API->>Mongo: Increment POI.ReplayQrScanCount
        API-->>Browser: InCooldown=true + message
    end

    Visitor->>Browser: Chọn ngôn ngữ và bấm nghe
    Browser->>API: POST /api/narration/play
    API->>Mongo: Count recent counted plays in 1 minute
    alt Dưới ngưỡng
        API->>Mongo: Insert NarrationLog Counted=true
        API->>Mongo: Increment POI.CountedTtsPlayCount
        API-->>Browser: logId + RateLimited=false
    else Quá ngưỡng
        API->>Mongo: Insert NarrationLog Counted=false, Replay
        API->>Mongo: Increment POI.ReplayTtsPlayCount
        API-->>Browser: logId + RateLimited=true
    end

    Browser->>Browser: Play audioUrl nếu có, fallback speech synthesis nếu không
    Browser->>API: POST /api/narration/finish(status, dwellTime)
    API->>Mongo: Update NarrationLog end state
Chapter 06

Mobile: nearby, geofence, offline queue

Luồng mobile khác WebQR ở chỗ nó phải chịu được điều kiện thực địa: chuyển mạng, GPS dao động và người dùng đang di chuyển.

sequenceDiagram
    actor User
    participant App as MAUI App
    participant PoiSvc as PoiService
    participant Track as GeofenceMonitorService
    participant API as TourGuide.API
    participant SQLite as OfflineStore

    User->>App: Mở app / Home
    App->>PoiSvc: GetPoisAsync()
    PoiSvc->>API: GET /api/poi/approved
    alt API reachable
        API-->>PoiSvc: Approved POIs
        PoiSvc->>SQLite: SavePoisAsync
        PoiSvc->>PoiSvc: FlushPendingEventsAsync
        PoiSvc-->>App: POI list
    else API fail
        PoiSvc->>SQLite: GetApprovedPoisAsync
        PoiSvc-->>App: Cached POI list
    end

    User->>App: Bật tracking/geofence
    App->>Track: StartAsync
    loop mỗi chu kỳ
        Track->>PoiSvc: SendPingAsync(location)
        alt Ping success
            PoiSvc->>API: POST /api/tracking/ping
        else Ping fail
            PoiSvc->>SQLite: Enqueue pending tracking_ping
        end
        Track->>PoiSvc: Refresh POIs nếu quá 5 phút
        Track->>Track: Tính khoảng cách tới POI và radius effective
        alt Enter geofence và chưa notify 30 phút
            Track-->>App: onPoiEntered(poi)
        end
    end

    User->>App: Quét QR hoặc mở detail
    App->>PoiSvc: RecordQrScan / StartNarration / FinishNarration
    alt Request fail
        PoiSvc->>SQLite: Enqueue pending event
    end

    User->>App: Có mạng trở lại
    App->>PoiSvc: FlushPendingEventsAsync
    PoiSvc->>API: replay từng pending payload
    API-->>PoiSvc: 200 OK
    PoiSvc->>SQLite: Delete pending event
Đây là chỗ mobile app có “độ sống” riêng: cache POI xuống SQLite, queue sự kiện thất bại và flush lại sau, chứ không phụ thuộc tuyệt đối vào online realtime.
Chapter 07

Admin dashboard aggregation và presence

Dashboard tổng hợp vừa dùng dữ liệu bền vững từ Mongo, vừa dùng presence cache trong memory để phản ánh online devices gần realtime.

sequenceDiagram
    actor Admin
    participant Portal as Admin Dashboard
    participant API as AnalyticsService + PresenceTracker
    participant Mongo as MongoDB

    Admin->>Portal: Mở /admin/dashboard
    par Overview load
        Portal->>API: GET /api/analytics/admin/overview
        API->>Mongo: Count owners, POIs, pending, approved
        API->>Mongo: Count counted QR logs và counted narration logs
        API->>Mongo: Read billing active + audit logs + trend data
        API-->>Portal: OverviewResponse
    and Heatmap load
        Portal->>API: GET /api/analytics/admin/heatmap?hours=4
        API->>Mongo: Load TrackingData in time window
        API-->>Portal: Heatmap points
    and POI overlay load
        Portal->>API: GET /api/poi/map-summary
        API->>Mongo: Load POIs
        API-->>Portal: POI points
    and Presence load
        Portal->>API: GET /api/tracking/online-devices
        API->>API: PresenceTracker.GetActive(threshold)
        API-->>Portal: Online devices
    end

    loop every 5 seconds
        Portal->>API: GET /api/tracking/online-devices
        API->>API: Refresh active list by LastSeenAt
        API-->>Portal: Latest online devices
        Portal->>Portal: Update online markers on map
    end
PresenceTracker là cache memory tại API instance, còn heatmap lại đến từ Mongo tracking history. Hai lớp này bổ trợ nhau: một lớp “ai đang online”, một lớp “ai đã di chuyển ở đâu”.
Appendix

Checklist khi review cùng team

Một số câu hỏi gợi ý nếu bạn dùng file này để review scope với PM, BA, backend và frontend.

Câu hỏi sản phẩm

  • Có muốn mở rộng merchant flow cho upload ảnh ngay trên form tạo/sửa không?
  • Có cần workflow xuất bản audio file thay vì chủ yếu dựa vào TTS fallback không?
  • Có giữ anti-spam hiện tại cho narration hay cần rule tinh hơn theo user/session?

Câu hỏi kỹ thuật

  • Nếu scale nhiều API instances, presence cache in-memory có cần external store không?
  • Translation cache có cần invalidation strategy sâu hơn khi source text đổi nhiều lần nhanh?
  • Billing webhook có cần idempotency logs rõ hơn ở tầng riêng không?
